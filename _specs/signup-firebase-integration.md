# Spec for signup-firebase-integration

branch: claude/feature/signup-firebase-integration
figma_component (if used): N/A

## Summary

Wire the existing `AuthForm` component (in `signup` mode) to Firebase Auth so that submitting the form creates a real Firebase account. After a successful signup, a randomly generated codename is assigned as the user's `displayName` and stored in a Firestore `users` document alongside the user's UID. Only the Firebase Web SDK (already initialised in `lib/firebase.ts`) may be used — no server-side calls or third-party libraries.

## Functional Requirements

- Calling `handleSubmit` on the signup form must call `createUserWithEmailAndPassword` from the Firebase Auth Web SDK using the `auth` export from `lib/firebase.ts`.
- On success, a codename must be generated by randomly picking one word from each of three distinct word sets and joining them in PascalCase (e.g. `SwiftCrimsonFox`). Each word set must contain unique, thematically distinct words.
- The generated codename must be set as the Firebase Auth user's `displayName` via `updateProfile`.
- A document must be created in the Firestore `users` collection (using the `db` export from `lib/firebase.ts`) with the fields `id` (the Firebase UID) and `codename`. The user's email must NOT be stored in this document.
- If signup fails (e.g. email already in use, weak password), an error message must be surfaced to the user in the form UI.
- While the async signup flow is in progress the submit button must be disabled to prevent duplicate submissions.
- The three word sets and the codename generation logic must live in a dedicated utility module (e.g. `lib/generateCodename.ts`) so they can be tested independently.
- Only the Firebase Web SDK (`firebase/auth`, `firebase/firestore`) should be imported — no Admin SDK.

## Possible Edge Cases

- Firebase returns an error code for an email already registered (`auth/email-already-in-use`) — display a user-friendly message rather than a raw error code.
- Firebase returns `auth/weak-password` — surface a readable message.
- Network failure mid-signup (auth succeeds but Firestore write fails) — the user is still created; the document write failure should be handled gracefully without breaking the UI.
- Codename collision in Firestore — out of scope for this feature; codenames are not required to be globally unique at this stage.
- The word sets should be large enough that the PascalCase combinations feel varied; aim for at least 10 words per set.

## Acceptance Criteria

- Submitting a valid email and password on the signup form creates a new Firebase Auth user.
- The created user's `displayName` matches a codename in the format `WordWordWord` (PascalCase, one word from each set).
- A document exists in the Firestore `users` collection with `id` equal to the new user's UID and `codename` matching `displayName`. No `email` field is present.
- Submitting with an already-registered email shows a friendly inline error (not a raw Firebase error code).
- The submit button is disabled while the request is pending.
- `generateCodename` is a pure function with no Firebase dependency and is covered by unit tests.

## Open Questions

- Should the three word sets have a theme consistent with the "heist" concept (e.g. adjectives, colours, animals)? animals

## Testing Guidelines

Create test file(s) in `./tests` mirroring the source structure. Focus on:

- `generateCodename` returns a string matching the PascalCase `WordWordWord` pattern.
- `generateCodename` always returns a word from each of the three sets (sample many calls and assert membership).
- `AuthForm` in `signup` mode calls `createUserWithEmailAndPassword` with the correct email and password on submit.
- After a successful Firebase response, `updateProfile` is called with the generated codename.
- After `updateProfile` resolves, the Firestore `addDoc` / `setDoc` call is made with `{ id, codename }` and no `email` field.
- A Firebase `auth/email-already-in-use` error results in a visible inline error message.
- The submit button is disabled while the async operation is pending and re-enabled after it settles.
